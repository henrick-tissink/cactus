**Goal:** Add a minimal root `package.json` that wires husky 9 + lint-staged so every `git commit` triggers Prettier (write) and ESLint (fix) on staged frontend files. Malformed commits are rejected locally; auto-fixable commits land with formatted content.

Frontend `package.json` stays independent — no npm workspaces, no shared dependencies.

**Tech Stack:** husky 9.x · lint-staged 15.x · Prettier (already installed in `src/frontend`) · ESLint (already installed in `src/frontend`)

- [ ] **Step 5: Scenario B — rejection verification**

Introduce an unfixable ESLint error. Pick a file that doesn't have its own ESLint problems, and add an unused variable.

Read the first lines of `src/frontend/src/main.tsx` to find a clean place to insert:

```bash
head -5 src/frontend/src/main.tsx
```

Now insert an unused variable. Use a portable approach — write a minimal change with `cat`:

```bash
cat > /tmp/main-patch.tsx <<'EOF'
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import './index.css';
import App from './App.tsx';

const _smokeUnused = 'should-fail-eslint';
EOF

# Append the rest of main.tsx (skip the first 4 import-style lines that we just rewrote — adjust by reading the actual file structure first)
```

Actually that's brittle. Use `sed` to inject one line at a known location instead. Or take the simplest approach — add the unused var on a new line at the top of the file body:

```bash
# Insert an unused variable on line 6 (after the imports)
sed -i.bak '5a\
const _smokeUnused = '"'"'should-fail-eslint'"'"';
' src/frontend/src/main.tsx
rm src/frontend/src/main.tsx.bak
```

(macOS/BSD `sed` requires the `.bak` suffix and a trailing newline after `\`. The `.bak` file is removed afterward.)

Verify the unused var is now present:

```bash
grep "_smokeUnused" src/frontend/src/main.tsx
```
Expected: prints the line with `_smokeUnused`.

Note: `_smokeUnused` is intentionally prefixed with underscore. The `@typescript-eslint/no-unused-vars` rule by default DOES still flag underscored names UNLESS the rule is configured with `argsIgnorePattern` / `varsIgnorePattern`. Check the existing config:

```bash
grep -A2 "no-unused-vars\|argsIgnorePattern\|varsIgnorePattern" src/frontend/eslint.config.js
```

If the config does NOT pattern-ignore underscored names (and `tseslint.configs.recommended` defaults don't), then `_smokeUnused` will trip the rule and ESLint can't auto-fix it (deletion isn't safe). Good — that's our unfixable error.

If the config DOES ignore underscored names, change the var name to `smokeUnused` (no underscore). Confirm by re-running grep on the rule.

## Post-execution notes (added 2026-05-06)

Two real plan defects surfaced during execution. Both were fixed at execution time and the implementation merged correctly; documenting here so the next person re-encountering this pattern doesn't have to re-derive the fixes.

- **Defect 1: bare `prettier`/`eslint` don't resolve from repo root.** The plan's lint-staged config invoked `prettier --write` and `eslint --fix` directly. lint-staged spawns commands using the repo-root `node_modules/.bin/` PATH, but those binaries live only in `src/frontend/node_modules/.bin/` — result: `[ENOENT]` on every commit. The "auto-discovery walks upward" claim in the spec was true for *config files* (Prettier and ESLint walk up from the file being processed), but binaries are PATH-resolved once at spawn time and don't walk anywhere. Fix: prefix every command with `npm exec --prefix src/frontend --` to add the frontend's bin dir to PATH for the spawned process. Final lint-staged config uses this for prettier (×2 globs) and eslint (×1).
- **Defect 2: ESLint v9 flat config doesn't auto-discover from cwd.** Even after fixing Defect 1, ESLint exited with `couldn't find an eslint.config.(js|mjs|cjs)`. The plan author was wrong about "ESLint walks upward from the linted file path" — that was the legacy `.eslintrc` behavior. ESLint v9 flat config search starts from `process.cwd()`, which for husky-invoked hooks is the repo root (no eslint.config.js there). Fix: pass `--config src/frontend/eslint.config.js` explicitly. Plugin imports inside that config still resolve relative to the config file's location, so `src/frontend/node_modules/...` is found correctly.
- **Branch ended up with 3 commits** instead of the plan's 2: the original `chore: add root package.json with husky + lint-staged` + `chore: add .husky/pre-commit running lint-staged` + a third `fix: pass --config to eslint so flat config resolves from repo root`. Squash-merge collapsed all three on `main` (PR #6, commit `827b9cc`).
- **Smoke Scenario A's "empty git commit" abort is correct lint-staged behavior, not a defect.** When `printf "\n  \n" >> ...` adds trailing whitespace + blank line and Prettier strips both, the net change is zero — lint-staged refuses to make an empty commit. This proves the hook fires and the commands succeed, even though no commit lands. A follow-up smoke that lands a real commit would need a substantive change Prettier reflows (e.g., a long line, a quote-style swap mid-string) — left as a future hardening note.
- **`.husky/_/` framework files are not committed** (auto-gitignored by husky's own `.husky/_/.gitignore` containing `*`). On a fresh clone or worktree, `npm install` at the repo root re-scaffolds the framework via husky's `prepare` script. Before that runs, `core.hooksPath = .husky/_` is set in `.git/config` but the framework files don't exist — git silently skips hooks. This is the "silent bypass" risk documented in the spec's Risks section. CONTRIBUTING.md (when one exists) should call out "run `npm install` at repo root once after cloning."
- **DRY refactor opportunity (Minor):** the lint-staged config repeats `npm exec --prefix src/frontend --` three times across the two globs. Acceptable verbosity for visibility today; if a fourth file-type entry is added later, migrate to `lint-staged.config.js` with a tiny helper function (`const fe = (cmd) => \`npm exec --prefix src/frontend -- ${cmd}\``).
- **Future-collision risk:** explicit `--config src/frontend/eslint.config.js` would override discovery if anyone adds a root-level `eslint.config.js` (e.g., to lint tooling JS). No root config exists today; flag for whoever adds one.
