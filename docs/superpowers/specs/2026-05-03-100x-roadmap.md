# 100x Roadmap — Cactus

**Created:** 2026-05-03 · **Last updated:** 2026-05-13
**Status:** Active, in execution. Phase 0 + Axis A + Axis O complete; product-wedge work (B, D, C, E, F, G) ahead.
**Iteration depth:** Mixed — solid on foundation axes (A, B, D), walking-skeleton on product axes (C, O, E, F, G).
**What "100x" means here:** the gap between *a thing that works* and *a thing high-earning South Africans choose over their bank app* — covering reliability, security, accurate spend visibility, AI assist, and an advisor distribution channel no other consumer-finance app has.

---

## Vision

Build the spending-visibility tool that high-earning South Africans actually use — accurate enough, automatic enough, and simple enough that a 30-second glance answers "where did the small money go last month?" — first for clients directly, then for the financial advisors who sit with those clients in annual reviews.

> **Interview thesis (2026-05):** the wedge isn't "help people budget." It's "tell people who don't need a budget where their small/discretionary money actually went." Lisa: *"I don't keep track at that level."* Sandile: *"the small stuff… when I look at the statement, the money disappears."* Existing bank-app tools (FNB, Investec) are set up but ignored because accuracy isn't high enough to drive behaviour. Win the accuracy bar and the rest follows.

Underneath that thesis, the program also has to make the foundation real: prod failures caught before users see them, data surviving a disk loss, bank feeds replacing manual entry, AI removing categorisation toil, every regression caught in CI before reaching the box.

This is a multi-week program covering 8 axes (A–G plus the mid-program-added Axis O) and a cleanup phase. It is **not** one design doc — each axis gets its own brainstorm → design → spec → plan → implementation cycle. This document is the program brief that sequences them.

### Axis B — Operations (Solid, 1–2 days)

**Goal:** Prod failures are caught by us, not users. Data survives the box.

**In scope**
- Sentry (or Glitchtip self-hosted) for frontend + backend errors. Source maps uploaded in CI.
- Structured JSON logging via Serilog → log host. Provider chosen during B's brainstorm (candidates: Better Stack, Axiom, Loki+Grafana self-hosted).
- Nightly `pg_dump` → Cloudflare R2 with 30-day retention. Restore tested as part of acceptance.
- Uptime monitor on `https://cactusmoney.app/health` (Better Uptime free tier or UptimeRobot). Email + push alert on >2 consecutive failures.
- Secrets management: GitHub Actions secrets → SSH → `/opt/cactus/.env` on deploy.
- Container restart policy already `unless-stopped`. Verify host auto-reboots if needed.

**Success criteria**
- Manually erroring a request appears in Sentry within 60 seconds with frontend source-mapped stack.
- Killing the API container raises an uptime alert within 5 minutes.
- Restoring a backup into a fresh Postgres container yields a working app: login succeeds, dashboard loads, seeded transactions visible.
- A monthly ops cost ceiling is set (budget value chosen during B's brainstorm).

**Out of scope**: APM tracing (defer); custom dashboards beyond uptime.

### Axis D — Backend hardening (Solid, 1–2 days)

**Goal:** Reduce attack surface and operator-visible noise. Prepare for sensitive E-axis work.

**In scope**
- Rate limiting via `AddRateLimiter` on `/api/auth/*`: login & register 5/min/IP, forgot-password 3/hour/email, all auth 30/min/IP global.
- Refresh-token rotation: each `/refresh` issues a new refresh token, old one revoked.
- Refresh-token revocation list (table) with cleanup job for expired entries.
- FluentValidation for all controller request DTOs.
- Request logging middleware producing structured fields (route, status, duration, user id, correlation id) — feeds into B's log shipping.
- Email verification enforced for sensitive actions (which actions decided in D's brainstorm).
- Migration deploy gating: advisory lock so concurrent restarts can't double-migrate.

**Success criteria**
- 11th login attempt within a minute returns 429.
- Replaying an old refresh token returns 401.
- Submitting malformed register body returns 400 with field-level errors (not 500).
- Every request log line has correlation id matching client header.

**Out of scope**: SSO/OAuth providers; password-less auth; passkeys (defer post-100x).

### Axis E — Product depth (Walking skeleton, decomposed)

This axis is itself a program. Each sub-axis becomes its own spec.

**E1 — Bank feeds (1–2 weeks)**
Goal: replace manual transaction entry with a live feed for at least one SA bank.

Interview signal (2026-05): two clients confirmed they track at account-balance level but not category level. Lisa: "I don't keep track at that level." Sandile: "the small discretionary stuff… when I look at the statement, the money disappears through all those small things." Existing bank-app budget tools (FNB, Investec) are set up but used only as fallback — accuracy isn't high enough to drive behaviour. **Accurate, automatic categorisation on real bank data is the wedge.**
Provider candidates (decided in E1's brainstorm): Stitch (SA-focused, supports FNB), TrueLayer (UK-leaning), DirectID, Yodlee.
Walking-skeleton: connect 1 bank, sync balances + transactions, surface in existing accounts UI. Webhook receiver with HMAC verification. No multi-bank, no historical backfill beyond 90 days, no auto-merge with manual entries (user resolves).

**E2 — Notifications (3–5 days)**
Goal: email notifs for goal milestones + weekly summary. Provider: Resend (recommended for SA + DX) or Postmark.
Walking-skeleton: 2 templates only. In-app preferences. No SMS, no push.

**E3 — Insights (1 week)**
Goal: surface 3–5 rule-based insights on the dashboard.
Examples: "Dining is 2× last month's average", "On track for emergency fund by Aug", "12 unclassified transactions need review".
Walking-skeleton: rules engine in `Cactus.Application`; cached daily; rendered as cards on Dashboard.

**E4 — Recurring detection (3–5 days)**
Goal: improve `transactions/recurring/detect` to flag missed payments and forecast next occurrence.
Walking-skeleton: enhance existing logic; surface "expected this week" + "missed this month" lists.

**Success criteria (axis-level)**
- A real user (you) connects FNB once and sees transactions appear within 10 minutes.
- At least one weekly insight email sends successfully.
- Insights cards render on the dashboard for the seeded test account.
- User-research bar (interview-validated): a user with bank feeds + insights can explain where their last-month discretionary spend went in under 30 seconds without opening their bank app. Measured by user interview, not CI.

**Out of scope**: receipts/OCR; budgeting wizard rewrite; investment account aggregation.

### Axis F — AI layer (Walking skeleton, decomposed)

**F1 — Auto-categorize (1 week)**
Goal: any uncategorized transaction gets a Claude-suggested category with confidence; user accepts/rejects in one click. Interview signal: the job is "tell me where my discretionary spend went, simply" — precision matters more than coverage. One wrong category on a dining row erodes trust faster than 10 uncategorised items, so low-confidence classifications should leave the transaction uncategorised (refuse to guess) rather than fill a guess for completeness.
Use Claude (Anthropic SDK) with prompt caching on the category list + user examples. Cost cap per user per month set in F1's spec.
Walking-skeleton: server-side classifier endpoint; UI button "auto-categorize 12 unclassified"; explicit user confirmation per suggestion.

**F2 — Personalised insights (1 week)**
Goal: weekly LLM-generated narrative summarising spending vs goals.
Walking-skeleton: cron job → render → email + dashboard panel.

**F3 — Chat interface (1–2 weeks)**
Goal: "How much did I spend on dining last month?" answered in natural language.
Tool-calling against existing API. Walking-skeleton: 5 read-only tools; no write actions; rate-limited per user.

**F4 — Household inflation rate (deferred, post-100x)**
Goal: once 6+ months of clean categorised data exists per user, derive a personalised CPI showing how this household's basket has shifted vs SA CPI. Huge for retirement planning — fundamentally changes the "how much do I need at 65" conversation, and is a strong hook for the Axis G advisor annual-review use case ("your personal inflation was 8% vs CPI 5% this year"). Out of program but flagged so it isn't lost. Depends on E1 + F1 producing clean, consistent categories over time; likely surfaces first inside the Axis G advisor view where retirement framing is most relevant.

**Success criteria (axis-level)**
- F1 categorizes ≥70% of typical SA bank transactions correctly on the seed dataset.
- F2 sends a weekly email that doesn't hallucinate numbers (grounded against API data).
- F3 answers 5 canned questions correctly.

**Out of scope**: model fine-tuning; image/receipt understanding; multi-turn task execution.

### Axis G — Advisor Portal (Walking skeleton, decomposed)

**Goal:** Turn financial advisors into a distribution channel by giving them visibility into client spend that no other tool provides. Interview signal (2026-05): advisors are actively adding spend questions to their annual review meetings (Paul is, in 2026) but have no shared tool to discuss line-item data with clients — and existing bank-app tools are inaccurate enough that they don't fill the gap. Sweet-spot target context: an advisor and client sitting together during an annual review, walking through categories on a shared screen.

**G1 — Consent + read-only client view (1–2 weeks)**
Goal: an advisor can list their consenting clients and view one client's expenses, category breakdown, and macro spend in the same shape the client sees.
Walking-skeleton: an `/advisor` route group with auth; advisor↔client link table created only via POPIA-compliant explicit client consent flow (specific, informed, voluntary, revocable, with named data scopes); read-only views of Transactions, Spending categories, Insights; audit-log table of who-viewed-what-when, queryable by both advisor and client.

**G2 — Advisor-led client onboarding (3–5 days)**
Goal: an advisor can invite a new client into Cactus via a link; the consent link is pre-established when the client completes Axis O onboarding.
Walking-skeleton: invite-link flow; client accepts, lands in existing onboarding, finishes with advisor already linked; no revenue-share / commission tracking.

**Success criteria (axis-level)**
- One advisor (Paul, real user) can view their list of consenting clients (≥2 in test) and drill into one client's last-month category spend in <5 clicks from advisor home.
- Consent is revocable: a client revokes → the advisor view returns 403 within 60s.
- Every advisor read is in the audit log; the client can view their own audit log from a settings page.

**Dependencies:** A (auth, tests), D (rate limiting + request logging — the audit trail piggybacks on D's correlation IDs), E1 (without real bank-feed data the advisor is just looking at the client's own self-classification, which is not a novel insight and breaks the pitch), E3 (insights — the artefact advisor and client discuss together).

**Out of scope (Axis G)**: advisor write actions or comments on client data; multi-advisor firms / hierarchies; commission tracking; cross-advisor client transfer; the "Household inflation rate" analytic (lives in F4); cross-policy aggregation across advisors who don't manage a given policy (huge scope, multi-FSP regulatory mess — flagged for follow-on program); structured advisor-led discovery forms / interview scripts.

## Sequence & dependencies

```
Phase 0  ─▶  A  ─┬─▶  B  ─┐
                 ├─▶  D  ─┼─▶  E1 ─┬─▶ E2/E3/E4 (any order) ─▶ F1 ─▶ F2/F3
                 └─▶  C  ─┘       └─▶ G1 ─▶ G2

                                                            [F4 deferred, post-100x]
```

- **A blocks everything** — no point doing B/C/D without tests + CI.
- **B/C/D can interleave** after A. Realistically (solo dev with me as executor) they run sequentially with light overlap; the diagram shows logical independence, not actual parallelism.
- **E1 (bank feeds) blocks F1** in usefulness — auto-categorization shines on real bank data, not seed data. F1 can technically run earlier on the seed set as a proof.
- **F2/F3 require B** for cost monitoring and observability of LLM calls.
- **G (advisor portal) branches off E1** — until bank feeds are live the advisor view is showing the client's own self-classification, which has no novel signal. G1 can run in parallel with E2/E3/E4 once E1 lands.

## Per-axis cycle (how each one happens)

For each axis (and each sub-axis under E and F):

1. Brainstorm in chat — clarifying questions, 2–3 approaches with tradeoffs, design walkthrough.
2. Spec written to `docs/superpowers/specs/YYYY-MM-DD-<axis-slug>-design.md`.
3. Self-review + user review of spec.
4. Implementation plan via `superpowers:writing-plans`, written to `docs/superpowers/plans/...`.
5. Implementation in a feature branch, PR, green CI, merge.
6. Post-merge verification on prod.

## Cross-cutting decisions (deferred to per-axis brainstorms)

These come up in multiple axes; each axis's brainstorm answers what it needs:

- **Hosting/cloud preferences**: stay on Hetzner+Cloudflare, or migrate to Fly/Render/AWS? (Affects A's CI deploy mechanics, B's monitoring choices.)
- **Log/metric stack**: SaaS (Better Stack, Axiom) vs self-hosted (Loki+Grafana). (B.)
- **Email provider**: Resend, Postmark, AWS SES. (E2, F2.)
- **Bank-feed provider**: Stitch, TrueLayer, DirectID. (E1.)
- **LLM provider**: Claude with prompt caching strongly recommended given pricing + thinking + tool-use story. (F1, F2, F3.)
- **POPIA / data residency**: SA-resident postgres? Encryption-at-rest beyond the disk? (D, E1, G.)
- **Advisor commercial model**: revenue share, flat per-client fee, free distribution with advisor paying for own seat? (G.)
- **Advisor portal auth model**: separate User-type or role-on-existing-User? Advisors typically have their own client investment accounts too, which leans toward role-on-existing-User. Decided in G1's brainstorm. (G1.)
- **Per-axis cost ceilings**: monthly $ caps for ops + LLM. Set during each axis's brainstorm.

## Risks & open questions

- **Solo executor velocity**: realistic calendar time is 2–3× the engineering days listed if the user is doing parallel real work. Mark explicitly in each axis spec.
- **Bank-feed regulatory**: Stitch / open-banking in SA require PoPIA awareness — flagged in E1.
- **LLM cost exposure**: F1's per-user-per-month cap prevents runaway spend; needs B's observability to enforce.
- **Single Hetzner box is still SPOF** even after B. A second box or move to a managed platform is out of program.
- **Goal tracking needs ongoing data, not just upfront capture** (interview signal, 2026-05): goal progress past day 1 is only meaningful with live balance/transaction feeds — withdrawals from an emergency fund to pay a car bill, market drift on investments, etc. Axis O's goal-pick is intent capture during onboarding (fine as-is, don't remove), but any *new* live-goal-progress UI is blocked on E1. Don't ship "live tracking" affordances on manual-entry data; the dishonesty will erode trust.
- **Advisor liability** (Axis G): financial advice is regulated in SA under FAIS. Showing an advisor a client's spend doesn't itself give advice, but the tool sits adjacent to advice — the boundaries of what advisors can and can't do *inside* the platform (annotations, recommendations, written notes) decided in G1's brainstorm with reference to the FAIS Code of Conduct.
